Querydsl の querydsl-sql

Posted on 2019-08-20 Edited on 2019-08-22 Views:

接触契机

由于客户要求, 将现有项目的数据库切换到要求的数据库. mybatis在mysql的基础上编写的sql具有针对性, 不适合移植, 为了更灵活的移植, 考虑到替换mybatis.
优先考虑到的持久层为hibernate.
hibernate完全符合本次的要求, 其完全基于对象关系映射, 避免直接操纵对应数据库的方言, 只要在开发的过程中避免native-sql的编写, 移植非常方便.
但考虑到其他方面的原因, 比如:
hql并不比使用sql简单。首先，从很多角度上看，hql等语言更加复杂和难以理解;其次就是性能上明显降低，速度更慢，内存占用巨大，而且还不好优化(对配置的人的专业知识要求比较高)。最为恼火的是，当关系的概念被替换为对象的概念之后，查询语言的灵活性变得很差，表达能力也比sql弱很多。写查询语句的时候受到各种各样的限制，一个典型的例子就是多表关联查询。
表之间的连接查询，被映射为实体类之间的关联关系，这样，如果两个实体类之间没有（实现）关联关系，你就不能把两个实体（或者表）join起来查询。
这是很恼火的事情，因为我们很多时候并不需要显式定义两个实体类之间的关联关系就可以实现业务逻辑，如果使用hql，只是为了join我们就必须在两个实体类之间添加代码，而且还不能逆向工程，如果表里面没有定义外键约束的话，逆向工程会把我们添加的关联代码抹掉。

基于这些考虑, 通过对比, 最后选定了querydsl.

Querydsl正式登场

接下来简单介绍querydsl-sql的一些配置和简单例子.

querydsl 简单介绍

querydsl 以流式api形式编写repository层代码, 有两种适用模式，querydsl-jpa和querydsl-sql。

querydsl-jpa和springdatajpa可以非常方便的集成，依据源码包里的@Entity注解的实体类生成查询实体，再调用JpaQueryFactory对应的api操作sql，绕开了hibernate在实体关系上的强硬配置。但这种最终发出的方言仍是hql，hql会使得在数据库验证语句效果不直观，而且jpa配置不自由。

最终, 还是选用的querydsl-sql.

querydsl-sql有个拓展，querydsl-sql-spring，集成了spring的事务，而且有自己的对象关系映射，映射依据是数据库表。也可以由实体覆盖表字段(具体待研究, NamingType)，配置非常灵活。
需要注意的是, 用了querydsl-sql-spring之后, 每个crud必须是有事务的, 否则会报错(Connection is not transactional).
在正式的使用querydsl做crud之前(调用提供的api), 需要按照querydsl的约定, 生成它需要的一系列实体. querydsl的所有对数据库的操作都由SQLQueryFactory去构建的. 通过不同的SQLTemplates(eg.MySQLTemplates, PostgreSQLTemplates, OracleTemplates…)解释成对应数据库的方言执行的.

说明:

  1. BaseDao : 公共接口, 包括一些公共查询方法, 如下(可以自行添加): BaseDao<T, ID>
    T : EntityType, 目前所有的实体类必须有主键, 且必须实现PrimaryEntity(querydsl codegen test已实现生成bean的时候继承);
    ID : primaryType.

  1. PrimaryEntity: 实体类的主键抽象接口.

    所有实体须实现PrimaryEntity, 实现getPrimaryKey()和setPrimaryKey()方法, 方法作用分别为返回实体的主键和设置实体的主键. (querydsl codegen test已实现生成bean的时候实现getPrimaryKey方法, 为了方便获取当前实体的主键, 增加了一个注解: @PrimayKey, codegen test生成实体bean的时候, 已加在主键列属性上.)

  2. AbstractBaseDao:

    AbstractBaseDao 的构造器为有参构造器, 为了根据spring配置的Datasource动态配置QueryDslConfig类(主要是配置根据不同的jdbcDriver来使用不同的SQLTemplate, 产生对应的SQLQueryFactory, QueryDslConfig.getInstance(dataSource) 单例获取queryDslConfig.), 如果再切换数据库只需要更改spring datasource配置就行, 简单方便.
    为了方便获取到主键对应的字段名, 定义了一个annotation – @PrimaryKey, 获取主键列的目的是为了集中实现BasDao里的公共接口.

    至此, 设计结束, 接下来根据表生成需要的实体类, 以及查询类(Querydsl操作数据库api需要的实体类型, 通常以实体名字前加Q的命名模式), 和dao接口+dao实现.

说明:

  1. codegen包里提供的主要生成bean的类为MetaDataExporter, 这里为了扩展, 新建了SimpleMetaDataExporter, 主要逻辑和MetaDataExporter一致, 加了一些额外的控制和功能:
    • 更改生成包的位置和包名, 可以自由配置.
    • 增加datasource(string)属性, 指定此属性以应用在不同的数据库上. (此项原因是因为MetaDataExporter在生成实体的时候需要根据数据库的表对应生成, 通过datasource取得DatabaseMetaData, 但通过DatabaseMetaData获取数据库表的时候, 在不同的数据库里需要的参数有差异.)
    • 增加cover(boolean) 属性, 判定如果已经存在该类是否需要覆盖.
    • 增加simpleDaoIfcSerializer(SimpleDaoIfcSerializer) 属性, 如果此属性不为空, 则按照simpleDaoIfcSerializer类定义的方式生成dao接口.
    • 增加simpleDaoImplSerializer(SimpleDaoImplSerializer) 属性, 如果此属性不为空(在增加simpleDaoIfcSerializer不为空的前提下), 则按照SimpleDaoImplSerializer类定义的方式生成dao接口的实现.
  2. MetaDataExporter 要生成某种类型的bean, 需要setSerializer, 默认有 beanSerializer属性, set之后生成实体bean(查询实体QBean默认就有, 不需要设置). 此处拓展了 beanSerializer – 拓展为SimpleBeanSerializer, 主要是为了给实体增加一些功能以便于在AbstractBaseDao中使用:
    • 实体都实现PrimaryEntity, 并重写实现getPrimaryKey()和setPrimaryKey()方法. 如果对应表无主键, 就不用实现PrimaryEntity, 同样也不会生成dao接口和实现.
    • 实体的主键上添加注解@PrimaryKey, 为了AbstractBaseDao 重写方法的时候使用.
  3. SimpleDaoIfcSerializer 实现 codegen.Serializer接口, 标明dao接口的生成内容.
  4. SimpleDaoImplSerializer 实现 codegen.Serializer接口, 标明dao实现的生成内容.

至此, 所需要的bean生成完成, 如果有需要其他的可额外添加. 接下来就可以愉快的crud了.
